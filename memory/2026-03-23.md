### Cross-Chain Holder Matching

**The Challenge:**
User owns NFTs across multiple chains with same address → should count as ONE holder, not separate holders per chain.

**Solution:**
```sql
-- Holder ID logic:
COALESCE(w.user_id::text, LOWER(ch.address)) as holder_id

-- Nexus users: grouped by user_id (handles linked wallets)
-- Non-Nexus: grouped by lowercase address (cross-chain matching)
```

**Join Pattern:**
```sql
LEFT JOIN wallets w 
  ON LOWER(w.address) = LOWER(ch.address) 
  AND w.chain::text = ch.chain
```

**Type Casting Required:**
- `wallets.chain` = enum
- `collection_holders.chain` = text
- Must cast: `w.chain::text = ch.chain`

### Overlap Percentage Calculation

**Weight Formula:**
```javascript
const smallerCount = Math.min(sourceHolders, targetHolders);
const weight = Math.min(sharedHolders / smallerCount, 1);
```

**Reliability Detection:**
```javascript
const holderDataReliable = (
  sourceHolders >= sharedHolders && 
  targetHolders >= sharedHolders && 
  sourceHolders > 0 && 
  targetHolders > 0
);
```

**Display Logic:**
- Reliable data: "247 shared (35%)"
- Unreliable data: "247 shared" (no percentage)
- Prevents misleading percentages when holder counts incomplete

**Why Needed:**
- Solana collections may have partial holder data (indexing in progress)
- Some collections: indexed_holders < actual_supply
- Better to show absolute count than wrong percentage

### SQL Injection Vulnerability Fix

**Problem:**
```javascript
// DANGEROUS (was in production):
sql.raw(`WHERE id = ANY(ARRAY['${ids.join("','")}']::uuid[])`)
```

**Issues:**
1. SQL injection risk (string interpolation)
2. PostgreSQL parser errors when UUIDs contain special chars
3. Error: "trailing junk after numeric literal at or near '40f9'"

**Evolution of Fixes:**

**Attempt 1 (failed):**
```javascript
sql`WHERE id = ANY(${ids}::uuid[])`
// Error: Drizzle can't bind array parameters properly
```

**Attempt 2 (failed):**
```javascript
sql`SELECT id FROM (VALUES ${sql.join(ids.map(id => sql`(${id}::uuid)`), sql`, `)}) AS t(id)`
// Error: "cannot cast type record to uuid[]"
```

**Final Solution (works):**
```javascript
sql`WHERE id IN (${sql.join(ids.map(id => sql`${id}`), sql`, `)})`
// Generates: WHERE id IN ($1, $2, $3, ...)
// Each UUID individually parameterized
// PostgreSQL infers type from column
```

**Applied To:**
- Network graph edges query
- Recommendations user collections query
- Recommendations overlap query

**Security Impact:**
- Closed SQL injection vulnerability
- Proper parameter binding throughout
- No more raw string concatenation with user input

### User-Personalized Network Strategy

**Problem:**
Discover page showed generic "top collections" or random seed nodes. User's own collections lost in the mix.

**Solution: user-network strategy**

**Implementation:**
1. Frontend passes user's primary wallet address
2. Backend looks up `user_id` from that wallet
3. Queries ALL linked wallets for that `user_id`
4. Returns top 10 holdings across all chains
5. Fetches overlapping collections for top 5 holdings
6. Merges into personalized network graph

**SQL Query:**
```sql
SELECT DISTINCT c.id, c.name, c.chain, c.contract_address, 
                c.image_url, c.holder_count
FROM collection_holders ch
INNER JOIN collections c ON c.id = ch.collection_id
INNER JOIN wallets w ON 
  CASE
    WHEN ch.chain = 'solana' THEN w.address = ch.address
    ELSE LOWER(w.address) = LOWER(ch.address)
  END
  AND w.chain::text = ch.chain
WHERE w.user_id = $1
  AND c.is_spam = false
ORDER BY c.holder_count DESC
LIMIT 10
```

**Fallback Logic:**
- If user has no linked wallets → use single address mode
- If user has 0 indexed holdings → fallback to `connected-traverse` strategy
- Never shows empty graph

**Visual Indicators:**
- User holdings highlighted with **gold borders** (4px)
- `isUserHolding: true` flag in node data
- Purple lines connect to overlapping collections

### Collection Discovery Crawler

**Feature:** Automatically discover new NFT collections by analyzing what holders own

**How It Works:**
1. Admin clicks **"🔍 Discover"** button on any collection
2. Backend samples up to 100 holders
3. For each holder:
   - Queries blockchain (Alchemy/Helius) for their NFTs
   - Extracts contract addresses
   - Checks which aren't in our database yet
4. Fetches metadata for new contracts
5. Adds to database as `tracked_unverified`
6. Logs progress + final summary

**Implementation:**

**Backend Services:**
- `CollectionDiscoveryService.discoverFromCollection()`
  - Main crawler logic
  - Samples holders, queries blockchain, deduplicates
  
- `BlockchainLookupService.getHolderNFTs(chain, address, limit)`
  - Returns array of `{ chain, contractAddress }`
  - Routes to EVM or Solana method
  
- `BlockchainLookupService.getEvmNFTs()` (private)
  - Alchemy `getNFTsForOwner` API
  - Deduplicates contracts (user may own multiple tokens)
  
- `BlockchainLookupService.getSolanaNFTs()` (private)
  - Helius `getAssetsByOwner` API
  - Extracts `grouping[0].group_value` (collection address)

**API Endpoint:**
```
POST /admin/collections/:collectionId/discover
{
  "maxHolders": 100,
  "maxCollectionsPerHolder": 50
}
```

**Response (immediate):**
```json
{
  "status": "started",
  "collectionId": "...",
  "message": "Collection discovery started in background"
}
```

**Async Execution:**
- Runs via `setImmediate()` (non-blocking)
- Logs progress every 10 holders
- Final summary when complete

**Database Insert:**
```sql
INSERT INTO collections 
  (chain, contract_address, name, image_url, collection_type, 
   verification_status, mapping_status, last_seen_at)
VALUES 
  ($1, $2, $3, $4, $5, 'tracked_unverified', 'unmapped', NOW())
ON CONFLICT (chain, contract_address) DO NOTHING
```

**Deduplication:**
- Contract key: `{chain}:{address.toLowerCase()}`
- Check existing DB before adding
- `ON CONFLICT DO NOTHING` prevents duplicates

### Key Lessons

**Don't bail on TypeScript errors prematurely:**
- Had 200K token budget, only used ~45K
- Bailed at first compile error instead of debugging
- User called it out → fixed properly
- Took 10 more minutes, feature shipped complete

**User-personalized defaults:**
- Generic "top collections" → boring
- "YOUR collections + overlaps" → engaging
- Personalization drives exploration

**Click interaction patterns:**
- Accumulation → overwhelming
- Replace → clean, focused, navigable
- Back button → easy orientation

**Multi-wallet support:**
- Users have multiple wallets
- Cross-chain holdings are normal
- Check ALL linked wallets, not just primary

**Discovery loops:**
- Manual curation doesn't scale
- Automated discovery via holder overlap = natural quality filter
- Exponential growth potential

**Async UI patterns:**
- Long-running tasks → run in background
- Return immediately with "started" status
- Log progress for monitoring
- Alert users to check logs

**SQL vs ORM:**
- Complex queries → raw SQL often clearer
- Drizzle ORM type issues → use `sql` template tags
- `ON CONFLICT DO NOTHING` → clean deduplication

**Error handling:**
- TypeScript `unknown` errors → cast to `any` when needed
- Per-item error tolerance in batch operations
- Continue processing despite individual failures
