### Cross-Chain Holder Matching

**The Challenge:**
User owns NFTs across multiple chains with same address → should count as ONE holder, not separate holders per chain.

**Solution:**
```sql
-- Holder ID logic:
COALESCE(w.user_id::text, LOWER(ch.address)) as holder_id

-- Nexus users: grouped by user_id (handles linked wallets)
-- Non-Nexus: grouped by lowercase address (cross-chain matching)
```

**Join Pattern:**
```sql
LEFT JOIN wallets w 
  ON LOWER(w.address) = LOWER(ch.address) 
  AND w.chain::text = ch.chain
```

**Type Casting Required:**
- `wallets.chain` = enum
- `collection_holders.chain` = text
- Must cast: `w.chain::text = ch.chain`

### Overlap Percentage Calculation

**Weight Formula:**
```javascript
const smallerCount = Math.min(sourceHolders, targetHolders);
const weight = Math.min(sharedHolders / smallerCount, 1);
```

**Reliability Detection:**
```javascript
const holderDataReliable = (
  sourceHolders >= sharedHolders && 
  targetHolders >= sharedHolders && 
  sourceHolders > 0 && 
  targetHolders > 0
);
```

**Display Logic:**
- Reliable data: "247 shared (35%)"
- Unreliable data: "247 shared" (no percentage)
- Prevents misleading percentages when holder counts incomplete

**Why Needed:**
- Solana collections may have partial holder data (indexing in progress)
- Some collections: indexed_holders < actual_supply
- Better to show absolute count than wrong percentage

### SQL Injection Vulnerability Fix

**Problem:**
```javascript
// DANGEROUS (was in production):
sql.raw(`WHERE id = ANY(ARRAY['${ids.join("','")}']::uuid[])`)
```

**Issues:**
1. SQL injection risk (string interpolation)
2. PostgreSQL parser errors when UUIDs contain special chars
3. Error: "trailing junk after numeric literal at or near '40f9'"

**Evolution of Fixes:**

**Attempt 1 (failed):**
```javascript
sql`WHERE id = ANY(${ids}::uuid[])`
// Error: Drizzle can't bind array parameters properly
```

**Attempt 2 (failed):**
```javascript
sql`SELECT id FROM (VALUES ${sql.join(ids.map(id => sql`(${id}::uuid)`), sql`, `)}) AS t(id)`
// Error: "cannot cast type record to uuid[]"
```

**Final Solution (works):**
```javascript
sql`WHERE id IN (${sql.join(ids.map(id => sql`${id}`), sql`, `)})`
// Generates: WHERE id IN ($1, $2, $3, ...)
// Each UUID individually parameterized
// PostgreSQL infers type from column
```

**Applied To:**
- Network graph edges query
- Recommendations user collections query
- Recommendations overlap query

**Security Impact:**
- Closed SQL injection vulnerability
- Proper parameter binding throughout
- No more raw string concatenation with user input

### SVG Rendering Bug Fix

**Problem:**
Collection images disappeared when focusing on different bubbles in network graph.

**Root Cause:**
```javascript
// Redrawing SVG added duplicate <defs> elements:
<defs>
  <clipPath id="clip-uuid1">...</clipPath>
  <clipPath id="clip-uuid1">...</clipPath> <!-- duplicate! -->
</defs>
// Browsers ignore duplicates → images fail to render
```

**Solution:**
```javascript
// Clear defs before each render, preserve <style>:
svg.selectAll('defs > *:not(style)').remove();
// Then add fresh gradients/clipPaths
```

**Effect:**
- Images persist across focus changes
- No duplicate ID warnings
- Smooth transitions between graph states

### Lessons Learned

**SQL parameterization with Drizzle:**
- Use `sql.join()` for IN clauses, not `ANY(array)`
- Each element gets individual parameter binding
- PostgreSQL infers types from column definitions
- Never use `sql.raw()` with string interpolation for user input

**SVG <defs> management:**
- Clear before each render to prevent duplicate IDs
- Preserve static elements like `<style>`
- Duplicate IDs cause silent rendering failures

**Data reliability detection:**
- Don't calculate percentages on incomplete data
- Add reliability flags to API responses
- Let frontend decide how to display

**Route architecture:**
- Establish canonical routes early
- Use redirects for legacy routes
- Don't duplicate features across multiple pages

**Admin UI patterns:**
- Provide direct access tools for edge cases
- Don't assume all data fits main navigation
- "Search by X" features unblock rare scenarios

**Holder indexing:**
- Collections exist before holder data
- Indexing must be triggered manually or via cron
- Clear UI feedback when data not yet available

**Force-directed graphs:**
- Iteration count affects performance
- Balance convergence vs. speed
- Different configs for different views (full vs. focused)

**Cross-chain identity:**
- Same address on different chains = same person (usually)
- User account linking is more reliable than address matching
- Support both: linked accounts + address matching
