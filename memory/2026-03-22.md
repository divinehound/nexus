# 2026-03-22 — NEXUS Development

## Major Migration: Reown AppKit (Unified Wallet Connections)

### Decision
Replaced RainbowKit + Solana Wallet Adapter with Reown AppKit for unified EVM + Solana wallet connections.

### Rationale
- User confusion with separate EVM and Solana buttons/modals
- Phantom wallet works with both EVM and Solana (old UI assumed Phantom = Solana only)
- No standard library existed that unified both ecosystems
- Reown AppKit claims to support both EVM and Solana in one interface

### Implementation
- **Removed:** RainbowKit, separate Solana adapter UI, dual-provider tree
- **Added:** `@reown/appkit`, `@reown/appkit-adapter-wagmi`, `@reown/appkit-adapter-solana`
- **New components:** 
  - `apps/web/src/lib/reown-config.ts` — wagmi adapter config
  - `apps/web/src/context/reown-provider.tsx` — unified provider with AppKit modal
  - `apps/web/src/components/wallet/connect-button-unified.tsx` — single button for all wallets
- **Updated:** Providers tree simplified to `ReownProvider > AuthProvider > children`

### Benefits
- One button opens one modal showing ALL wallets (EVM + Solana)
- No more confusing tab-based UI
- Smaller bundle size
- Better wallet compatibility

### Required Configuration
- **Env var:** `NEXT_PUBLIC_WALLETCONNECT_PROJECT_ID` (get from https://cloud.reown.com)
- Graceful fallback: builds without it, shows console warning at runtime
- Added to `.env.example`

## Wallet Linking UX: Modal-Based Ephemeral Connections

### Old Flow (Broken)
- Embedded wallet connect buttons on `/me` page
- Required persistent wallet connection to link
- Confusing which provider to use (Phantom = Solana assumption)
- Had to disconnect/reconnect to link different wallets

### New Flow (Proper)
1. Click "Link New Wallet" button
2. Modal opens with both EVM and Solana sections
3. Connect any wallet (ephemeral)
4. Green box shows "Connected: 0x123..."
5. Click "Link This Wallet"
6. Sign message
7. Wallet disconnects automatically
8. Linked wallet saved to account

### Implementation
- `apps/web/src/components/wallet/link-wallet-modal.tsx` — modal-based linking
- Same UX pattern applied to sign-in flow (`connect-button-v2.tsx`)
- Ephemeral connections: wallet disconnects after auth/linking
- JWT bearer token is the session, not wallet connection state

- `1b174aa` — Modal-based wallet linking
- `0f314d6` — Unified auth flow (matching link wallet UX)

## Technical Context

### Wallet Connection Ecosystems
- **EVM wallets:** Use WalletConnect protocol, SIWE (Sign-In With Ethereum)
- **Solana wallets:** Different protocol, different signing methods
- **No unified standard existed** before Reown AppKit
- RainbowKit (EVM only) and Solana Adapter (Solana only) were separate ecosystems

### Why Reown AppKit
- First library to claim unified EVM + Solana support
- WalletConnect recently rebranded to "Reown"
- AppKit is their new unified interface
- Stack: AppKit (UI) → wagmi (React hooks) → WalletConnect/Reown (protocol)

## Architectural Lessons

### Modal UX Patterns
- **Always lock body scroll** when modal is open
- **Max height constraint** prevents bottom cutoff on mobile
- **Scrollable content area** with fixed header
- **Responsive padding** (less on mobile, more on desktop)
- **Visual separation** (border between header and content)

### Wallet Connection Best Practices
- **JWT bearer token = session**, not wallet connection state
- **Ephemeral connections** for auth/linking (disconnect after signing)
- **Don't assume wallet = chain** (Phantom works with EVM and Solana)
- **Unified UI** reduces cognitive load vs. separate tab-based flows

### Migration Strategy
- **Graceful fallbacks** for missing env vars (build succeeds, runtime warning)
- **Provider simplification** when consolidating ecosystems
- **Keep old components** until new flow verified in production
