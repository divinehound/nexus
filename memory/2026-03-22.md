# 2026-03-22 — NEXUS Development

## Major Migration: Reown AppKit (Unified Wallet Connections)

### Decision
Replaced RainbowKit + Solana Wallet Adapter with Reown AppKit for unified EVM + Solana wallet connections.

### Rationale
- User confusion with separate EVM and Solana buttons/modals
- Phantom wallet works with both EVM and Solana (old UI assumed Phantom = Solana only)
- No standard library existed that unified both ecosystems
- Reown AppKit claims to support both EVM and Solana in one interface

### Implementation
- **Removed:** RainbowKit, separate Solana adapter UI, dual-provider tree
- **Added:** `@reown/appkit`, `@reown/appkit-adapter-wagmi`, `@reown/appkit-adapter-solana`
- **New components:** 
  - `apps/web/src/lib/reown-config.ts` — wagmi adapter config
  - `apps/web/src/context/reown-provider.tsx` — unified provider with AppKit modal
  - `apps/web/src/components/wallet/connect-button-unified.tsx` — single button for all wallets
- **Updated:** Providers tree simplified to `ReownProvider > AuthProvider > children`

### Benefits
- One button opens one modal showing ALL wallets (EVM + Solana)
- No more confusing tab-based UI
- Smaller bundle size
- Better wallet compatibility

### Required Configuration
- **Env var:** `NEXT_PUBLIC_WALLETCONNECT_PROJECT_ID` (get from https://cloud.reown.com)
- Graceful fallback: builds without it, shows console warning at runtime
- Added to `.env.example`

## Technical Context

### Wallet Connection Ecosystems
- **EVM wallets:** Use WalletConnect protocol, SIWE (Sign-In With Ethereum)
- **Solana wallets:** Different protocol, different signing methods
- **No unified standard existed** before Reown AppKit
- RainbowKit (EVM only) and Solana Adapter (Solana only) were separate ecosystems

### Why Reown AppKit
- First library to claim unified EVM + Solana support
- WalletConnect recently rebranded to "Reown"
- AppKit is their new unified interface
- Stack: AppKit (UI) → wagmi (React hooks) → WalletConnect/Reown (protocol)

## Wallet Linking Mobile: Dedicated /link-wallet Page

### Problem
Mobile wallet apps (e.g., Base app) open in their own browser context. Cookies/sessions from Chrome don't transfer. Linking flow broken.

### Solution
Dedicated `/link-wallet` page with JWT token in URL:
- `/link-wallet?token=JWT_TOKEN`
- No navbar (prevents duplicate sign requests)
- Manual step-by-step flow (removed auto-sign)
- User controls all actions explicitly

### Implementation
- `apps/web/src/app/link-wallet/page.tsx` — dedicated linking page
- `apps/web/src/app/link-wallet/layout.tsx` — no navbar layout
- `apps/web/src/components/wallet/link-wallet-button.tsx` — links to page with token
- Process address tracking via `useRef` to prevent duplicate operations

### Key Lessons
- Mobile wallet apps cause browser context switching
- Context switching invalidates nonces and creates race conditions
- Single nonce request required per authentication attempt
- User must control timing to avoid context issues
- **No auto-magic** — eliminates race conditions

## SIWE Authentication Fixes

### Issue 1: Malformed SIWE Message
**Error:** "Malformed SIWE message"

**Problem:** Template literal format wasn't matching strict SIWE spec requirements.

**Fix:** Use `array.join('\n')` for exact newline control:
```javascript
const message = [
  `${domain} wants you to sign in with your Ethereum account:`,
  address,
  '',
  'Sign in to NEXUS',
  '',
  `URI: ${origin}`,
  'Version: 1',
  `Chain ID: ${chainId}`,
  `Nonce: ${nonce}`,
  `Issued At: ${timestamp}`
].join('\n');
```

**Commit:** `873de17` — Strict SIWE format with array join

### Issue 2: Wrong Chain ID (Always 1 = Ethereum)
**Error:** Signing with Base wallet but message showed "Chain ID: 1"

**Problem:** `caipNetworkId` wasn't available or formatted differently.

**Fix:** Call `eth_chainId` RPC method directly:
```javascript
const chainIdHex = await evmProvider.request({ method: 'eth_chainId' });
const chainId = parseInt(chainIdHex, 16);
```

**Result:** Base wallet correctly shows "Chain ID: 8453"

**Commits:**
- `897d6e5` — Get chain ID from eth_chainId RPC
- `2f29586` — Use actual chain ID in SIWE message

### Issue 3: Invalid Nonce
**Error:** "Invalid nonce"

**Problem:** Mobile browser context switching caused new nonce requests, invalidating previous nonce.

**Fix:** Remove auto-sign behavior, add manual "Sign In" button:
- User clicks "Connect Wallet" → wallet connects
- Green "Sign In" button appears
- User clicks when ready → single nonce request
- Signs → logs in

**Commit:** `90227a7` — Replace auto-sign with manual "Sign In" button

### SIWE Format Reference
```
{domain} wants you to sign in with your Ethereum account:
{address}

Sign in to NEXUS

URI: {origin}
Version: 1
Chain ID: {chainId}
Nonce: {nonce}
Issued At: {timestamp}
```

**Critical:**
- Blank lines after address and message text
- Chain ID must match wallet's current chain
- Nonce must be fresh and valid
- No extra whitespace or formatting

## Wallet Transfer Confirmation Flow

### Problem
When linking a wallet already linked to another user, API returns 409 Conflict with `confirmationToken`. Need to confirm transfer.

### Flow
1. User tries to link wallet
2. API returns: `{ error: 'wallet_already_linked', confirmationToken: '...' }`
3. UI shows confirmation dialog
4. User clicks "Yes, transfer this wallet"
5. Client sends: `POST /me/wallets/move` with `{ confirmationToken, signature, message }`
6. Wallet transferred to current user

### Implementation
- Parse `err.data` (not `err.error`) for ApiError response body
- Extract `err.data.error` and `err.data.confirmationToken`
- Show confirmation UI
- Call `/me/wallets/move` endpoint

### API Structure
**ApiError class:**
- Properties: `status`, `statusText`, `data`
- Response body stored in `data` property
- Access fields: `err.data.error`, `err.data.confirmationToken`, `err.data.message`

## Lessons: Data Completeness Challenge

### Problem Identified
Current data: only holdings for users who signed in (hundreds of wallets)
Full ecosystem: millions of NFT holders
Coverage: <0.01%

**For accurate overlap analysis, need:**
- Full holder lists for analyzed collections
- OR representative sample

### Options Evaluated

1. **Accept Limited Scope** (Current)
   - "Among NEXUS users, what overlaps?"
   - ✅ No additional work
   - ✅ Still valuable for community
   - ❌ Incomplete picture
   - ❌ Biased toward early users

2. **Index Popular Collections Fully** (Recommended)
   - Index ALL holders for verified/featured collections
   - ✅ Accurate for important collections
   - ✅ Best user experience
   - ❌ API costs (~$200/month for top 100)
   - ❌ Storage costs (minimal)
   - ❌ Time to index

3. **Hybrid: Full + Sample** (Best Balance)
   - Full index for verified collections (top 100-500)
   - Random sample (10-20%) for others
   - User holdings always fully indexed
   - ✅ Accurate for important collections
   - ✅ Directional data for long-tail
   - ✅ Personalized for logged-in users

4. **On-Demand API Queries** (No Storage)
   - Query Reservoir/Alchemy when viewing collection
   - ✅ No upfront indexing
   - ✅ Always fresh
   - ❌ Slow page loads
   - ❌ API rate limits
   - ❌ Can't pre-compute network graphs

### Decision
**Start with Current Data, Build Toward Hybrid**

**Immediate (Shipped):**
- Build overlap analysis using current user data
- Be transparent: "Based on N NEXUS users"
- Still valuable — shows taste clusters in community

**Near-term (Stage 1.3):**
- Index full holder lists for verified collections
- Top 100 by volume/holders
- Collections with >10 NEXUS users
- Show data quality indicator on UI

**Long-term (Stage 4.2):**
- Auto-index when collection hits thresholds (25+ holders, verified, featured)
- Incremental updates (don't re-index from scratch)
- Cost monitoring dashboard

### Implementation Plan
1. Add `index_status` enum to collections table: `'nexus_only' | 'sampled' | 'full'`
2. Background job to index verified collections
3. UI shows badge: "Full Data" vs "NEXUS Users Only"
4. Admin dashboard to trigger manual indexing
